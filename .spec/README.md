# `.spec`

This folder is the Spec Led Development layer for `specled_ex`.

<!-- covers: spec.workspace.readme_present -->

This project uses its own tooling to specify itself.

## Layout

- `README.md` — this file (authored)
- `AGENTS.md` — local operating guidance for agents working in `.spec/` (authored)
- `decisions/README.md` — ADR guidance for durable cross-cutting decisions (authored)
- `decisions/*.md` — accepted or superseded ADRs for the current workspace (authored)
- `specs/spec_system.spec.md` — meta-spec for this workspace (authored)
- `specs/package.spec.md` — package-level contract (authored)
- `specs/index_state.spec.md` — index building and canonical state rules (authored)
- `specs/block_schema.spec.md` — structured block schema rules (authored)
- `specs/parser.spec.md` — parser module spec (authored)
- `specs/verification.spec.md` — verification workflow spec (authored)
- `specs/mix_tasks.spec.md` — Mix task workflow spec (authored)
- `specs/decision_governance.spec.md` — ADR parsing and validation rules (authored)
- `specs/reporting.spec.md` — reporting and coverage summary rules (authored)
- `specs/diffcheck.spec.md` — diff-aware co-change validation rules (authored)
- `state.json` — generated by `mix spec.verify`

## Workflow

1. Update current-truth subjects in `specs/`.
2. Update `decisions/*.md` only when a cross-cutting policy should stay durable.
3. Use Git history and pull requests as the time dimension for how a change evolved.
4. Run `mix spec.verify --debug` to validate structure and references.
5. Run `mix spec.check` for the strict conformance gate.
6. Run `mix spec.diffcheck` when code, docs, or tests changed.
7. Use `mix spec.report` when you want coverage and weak-spot summaries.

## Notes

- Structured blocks prefer YAML, though JSON also parses.
- Verification kinds use `source_file`, `test_file`, etc. (not `test` or `doc`)
- Subject specs may reference ADR ids through `spec-meta.decisions`.
